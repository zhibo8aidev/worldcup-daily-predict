# Technical Notes - 世界杯每日赛程预测活动

## 1. 当前技术判定
- 前端形态：H5，嵌入 App WebView
- 前端设计目标：mobile-web
- 前端实现建议：单页活动架构 + 模块化业务组件
- 后端策略：在“接口是否已有”未最终确认前，默认准备前端数据适配层 + 可新增接口的技术方案

## 2. 建议拆分的能力域
### 2.1 活动前端
- 首页 / 榜单页 UI 与交互
- 比赛卡片状态机
- 日期切换与列表渲染
- 登录唤起 / 比赛详情跳转 / WebView 桥接封装

### 2.2 预测业务接口
- 获取比赛日列表与赛程详情
- 获取用户个人战绩摘要
- 提交单题预测
- 获取投票分布
- 获取达人榜（日榜 / 总榜）
- 获取赛后结算结果

### 2.3 配置与结算
- 球员焦点题手工配置表
- 预测截止判定
- 比赛结算与异常态处理
- 排行榜刷新策略

## 3. 当前明确的产品变化
- 分享战绩卡取消，前端无需实现长图生成与分享面板
- 预测项按单题即时提交，不做整场统一提交
- 球员题先由开发手工维护配置，不强依赖后台 CMS

## 4. 仍需最终确认的技术点
- 赛程、预测、排行、结算等接口是否已有线上服务
- 若已有接口，字段格式、鉴权方式、错误码与限流策略是什么
- App 提供哪些 JSBridge 能力（登录、分享、路由跳转、埋点）
- 比赛详情按钮的 URL / schema 规则

## 5. 设计落地注意事项
- KV 底色与页面背景需无缝衔接，避免滚动断层
- 顶部模块需带负边距搭叠效果
- 预测按钮必须保留高质感渐变与微立体交互反馈
- 低赛程日需通过底部环境渐变或扩展文案撑满视觉重心

## 6. 下一阶段建议
- 若用户认可 proposal/specs，可进入 PRD review
- PRD 通过后，启动设计阶段，基于当前 UI 规范产出 HTML 高保真 mockups 和 design.md
- 设计评审通过后，再进入 tasks 拆解与 ACP 编码执行
