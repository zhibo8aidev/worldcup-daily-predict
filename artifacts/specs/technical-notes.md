# Technical Notes - 世界杯每日赛程预测活动

## 1. 当前技术判定
- 前端形态：H5，嵌入 App WebView
- 前端设计目标：mobile-web
- 前端实现建议：单页活动架构 + 模块化业务组件
- 后端策略：在“接口是否已有”未最终确认前，默认准备前端数据适配层 + 可新增接口的技术方案

## 2. 已确认的 PRD 变化
- 分享战绩卡重新纳入范围
- 个人战绩摘要增加用户头像展示
- 达人榜入口改为右上角悬浮按钮
- 活动规则入口也位于右上角悬浮区
- 比赛详情跳转入口改为点击对阵头像或比赛时间区域
- 球员焦点题继续手工维护

## 3. 当前最大阻塞
### 3.1 提交模型冲突
PRD 当前同时存在两种互斥提交方式：
- **整场统一提交**：三题全部完成后点击“提交预测”
- **单题独立提交**：可只预测 1 题或 2 题

### 3.2 受影响范围
- 前端状态机
- 提交接口设计
- 本地暂存逻辑
- 结算维度
- 排行口径
- 埋点触发时机
- 分享卡统计口径

## 4. 技术影响判断
如果采用 **整场统一提交**：
- 前端需要为每场比赛维护“草稿态”
- 提交按钮成为关键交互节点
- 提交接口更偏向 match-level payload
- 未填全时需要校验和 Toast

如果采用 **单题独立提交**：
- 前端可即时提交
- 接口更偏向 question-level payload
- 投票分布可在每题提交后即时展示
- 排行与统计逻辑要按题粒度累积

## 5. 设计状态判断
- 当前已产出的 design.md 和 HTML mockups 基于旧版 PRD
- 由于提交模型与页面结构已发生变化，现有设计稿不能继续直接进入审批
- 需要在 PRD 冲突澄清并重新确认后，再重开 design 阶段更新设计产物

## 6. 仍需最终确认的技术点
- 提交模型到底选哪一种
- 赛程、预测、排行、结算、分享等接口是否已有线上服务
- 若已有接口，字段格式、鉴权方式、错误码与限流策略是什么
- App 提供哪些 JSBridge 能力（登录、分享、路由跳转、埋点）
- 比赛详情按钮的 URL / schema 规则

## 7. 下一阶段建议
- 先澄清提交模型冲突
- 更新 proposal/specs 后重新进入 prd_review
- PRD 再次批准后，重新生成 design.md + HTML mockups
- 设计评审通过后，再进入 tasks 拆解与 ACP 编码执行
